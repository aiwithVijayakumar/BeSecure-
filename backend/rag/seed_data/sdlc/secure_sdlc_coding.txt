Secure SDLC & Secure Coding Guidelines - Knowledge Base

=== SECURE SDLC PHASES ===

--- Phase 1: Requirements ---
Security requirements must be defined before development begins.
- Define authentication and authorization requirements
- Identify data classification (PII, financial, health data)
- Define encryption requirements for data at rest and in transit
- Specify logging and monitoring requirements
- Define incident response requirements
- Risk assessment must be completed before design begins

--- Phase 2: Design ---
Security architecture must be reviewed before implementation.
- Threat modeling must be performed (STRIDE methodology)
- Security controls must be designed for each identified threat
- Principle of least privilege must be applied to all components
- Data flow diagrams must identify all data touchpoints
- API security design including authentication mechanism
- Key management strategy must be defined

--- Phase 3: Development / Coding ---
Secure coding standards must be followed.
INPUT VALIDATION:
- Validate ALL input server-side; never trust client-side alone
- Use allowlists over denylists for input validation
- Validate data type, length, format, and range
- Reject input that fails validation; do not sanitize and accept
- Apply contextual encoding for output (HTML, SQL, XML, JSON)

AUTHENTICATION:
- Implement MFA for all sensitive operations
- Use strong password policies (minimum 12 chars, complexity)
- Implement account lockout after failed attempts
- Use secure session management (HttpOnly, Secure cookies)
- Never store plain-text passwords; use Argon2, bcrypt, or PBKDF2

AUTHORIZATION:
- Enforce RBAC on every endpoint
- Validate authorization server-side on every request
- Never rely on client-supplied roles or permissions
- Apply IDOR protections (use indirect object references)

CRYPTOGRAPHY:
- AES-256-GCM for symmetric encryption
- RSA-2048+ or ECC P-256+ for asymmetric encryption
- SHA-256 or SHA-3-256 for hashing (never MD5 or SHA-1 for security)
- PBKDF2, bcrypt, or Argon2 for password hashing
- Use cryptographically secure random number generators
- Never implement custom cryptography

SQL:
- Use parameterized queries / prepared statements exclusively
- Use ORM where possible (Room for Android)
- Never concatenate user input into SQL queries
- Apply principle of least privilege to database accounts

ERROR HANDLING:
- Implement centralized error handling
- Never expose stack traces or internal errors to users
- Log errors server-side with sufficient detail for investigation
- Return generic error messages to clients (avoid information disclosure)

LOGGING:
- Log all authentication events (success and failure)
- Log all authorization failures
- Log all input validation failures
- NEVER log: passwords, tokens, PII, credit card numbers, health data
- Ensure logs are tamper-resistant

--- Phase 4: Testing ---
Security testing must be performed before release.
- Static Application Security Testing (SAST) mandatory
- Dynamic Application Security Testing (DAST) for web components
- Dependency vulnerability scanning (OWASP Dependency-Check)
- Penetration testing for new features or major changes
- Security code review by qualified reviewers
- All critical/high findings must be remediated before release

--- Phase 5: Deployment ---
Secure configuration must be applied before production deployment.
- Remove all debug flags and test credentials before release
- Apply network security configuration (deny cleartext)
- Enable ProGuard/R8 obfuscation
- Configure proper app signing
- Review and restrict all exported components
- Disable android:debuggable in production
- Set android:allowBackup=false unless backup is required

--- Phase 6: Maintenance ---
Ongoing security monitoring and response.
- Monitor for new CVEs in dependencies
- Implement security patch management process
- Conduct regular security assessments
- Establish and test incident response plan
- Provide security training to development team

=== SECURE CODING VIOLATIONS ===

CRITICAL VIOLATIONS (Block Release):
1. Hardcoded credentials or API keys in source code
2. SSL/TLS certificate validation disabled
3. SQL injection vulnerabilities
4. Plaintext storage of passwords or sensitive tokens
5. Root checks absent in security-sensitive apps

HIGH VIOLATIONS (Fix Before Release):
1. Weak cryptographic algorithm (MD5, DES, RC4)
2. Insecure data storage (unencrypted SharedPreferences with sensitive data)
3. WebView with JavaScript enabled + addJavascriptInterface
4. Exported components without permissions
5. Debuggable flag enabled in production
6. Cleartext network traffic allowed

MEDIUM VIOLATIONS (Fix in Next Sprint):
1. Missing certificate pinning
2. Logging of sensitive data
3. Use of java.util.Random instead of SecureRandom
4. Backup enabled with sensitive data
5. Overly broad permission requests
6. Missing anti-tampering controls

LOW VIOLATIONS (Track and Address):
1. Older but non-broken algorithms in non-security contexts
2. Missing code obfuscation
3. Missing root detection
4. Missing emulator detection
5. Overly verbose error messages in debug builds

=== ANDROID-SPECIFIC SECURE CODING CHECKS ===

1. AndroidManifest Review:
   - android:debuggable must be false in release
   - android:allowBackup should be false
   - android:usesCleartextTraffic should be false
   - All exported components should require permissions
   - Only necessary permissions should be declared

2. Data Storage:
   - Prefer internal storage over external storage for sensitive data
   - Use EncryptedSharedPreferences (Jetpack Security) for sensitive preferences
   - Use SQLCipher for encrypted SQLite databases
   - Never store sensitive data in world-readable files
   - Use Android Keystore for cryptographic keys

3. Network Security:
   - Define Network Security Config to restrict cleartext
   - Implement certificate pinning for high-security connections
   - Use OkHttp with proper TLS configuration
   - Validate server certificates; never override errors to proceed

4. IPC (Inter-Process Communication):
   - Define explicit permissions on all exported components
   - Validate all incoming Intent data
   - Use PendingIntent flags properly (FLAG_IMMUTABLE)
   - Restrict Content Provider access with READ/WRITE permissions
   - Avoid implicit broadcasts for sensitive data

5. WebView Security:
   - Disable JavaScript unless absolutely necessary
   - Disable file access (setAllowFileAccess=false)
   - Disable universal file access (setAllowUniversalAccessFromFileURLs=false)
   - Validate all URLs before loading
   - Audit all addJavascriptInterface methods

6. Cryptography:
   - Use AES/GCM/NoPadding for symmetric encryption
   - Generate unique IV for each encryption
   - Use Android Keystore for key storage
   - Avoid ECB mode
   - Use SecureRandom for all random values
